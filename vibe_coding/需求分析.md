
# **个人技术知识引擎 (Personal Tech Knowledge Engine) - V1.0 需求规格说明书**

*   **版本:** 1.0
*   **日期:** 2025年7月13日
*   **制定人:** 资深AI研发架构师 (AI R&D Architect) & 项目发起人欧阳宇

## 1. 项目愿景 (Project Vision)

构建一个结构化的个人知识管理系统，该系统旨在对AI领域的关键技术（以论文和代码为载体）进行深度、可追溯、可复用的分析。它不仅是一个学习笔记仓库，更是一个训练**“算法-系统”双螺旋思维**、连接理论与实践、并能产出高层次技术洞察的个人知识引擎。其最终产出物，旨在达到专业投研报告的深度和严谨性，成为个人技术能力最有力的证明。

## 2. 核心原则 (Core Principles)

所有分析工作必须严格遵守以下三大原则：

*   **原则一：可验证性原则 (Principle of Verifiability)**
    *   **描述:** 每一个断言都必须有明确的引用来源。禁止任何无根据的猜测或模糊表述。
    *   **执行标准:** 理论观点需注明论文章节，代码逻辑需注明文件路径与行号，实验数据需注明来源表格。

*   **原则二：静态穿透原则 (Principle of Static Penetration)**
    *   **描述:** 在**不配置和运行代码**的前提下，通过静态代码分析、逻辑推理和模式识别，洞察其设计意图和潜在性能特征。
    *   **执行标准:** 能够逻辑上追踪核心变量的变换路径；能够识别理论上的重操作（如`einsum`, `roll`）并预判其为潜在性能瓶颈；能够分析项目依赖并识别特殊的软硬件需求。

*   **原则三：“展示，而非描述”原则 (Principle of "Show, Don't Tell")**
    *   **描述:** 优先使用图、表、代码块、公式等可视化元素来传递信息，而非大段的文字描述。
    *   **执行标准:** 结构用架构图，算法用伪代码，结果用表格，关键代码用代码块加注释。

## 3. 系统架构 (System Architecture)

本知识引擎采用双层架构：

*   **3.1. 深度分析层 (Deep Dive Layer)**
    *   **架构模式:** **“卷宗式”架构 (Case File Architecture)**。每一项技术（一篇论文）都是一个独立的“卷宗”（目录），包含对其完整、深入的分析。
    *   **目标:** 实现对单点技术的深度穿透。

*   **3.2. 横向对比层 (Comparative Layer)**
    *   **架构模式:** **手动触发的“结构化对比分析”**。由用户手动指定多个已分析过的“卷宗”，并基于标准化的对比框架，生成一份独立的对比分析报告。
    *   **目标:** 实现跨技术点的横向连接，生成高层次洞察。

## 4. 核心功能与产物规格 (Core Features & Deliverable Specifications)

### 4.1. 深度分析层：“卷宗”规格

每个“卷宗”为一个独立目录，其标准结构与内容模板如下：

#### **目录结构**
```
/MAN_OF_INTEREST_paper_name   # 例如: /swin_transformer_v1
|
|-- paper.pdf                     # 原始论文
|-- /src_code/                      # 官方/复现代码库 (只读)
|
|-- 1_SUMMARY.md                    # 概览与一句话结论
|-- 2_THEORY_DECONSTRUCTION.md      # 理论层拆解
|-- 3_ENGINEERING_ANALYSIS.md       # 工程与代码实现分析 (静态)
|-- 4_CRITIQUE_AND_REFLECTION.md    # 批判性思考与个人启发
```

#### **文件内容模板 (质询式模板 V2.1)**

**`1_SUMMARY.md`**
*   **论文信息:** Title, Authors, Conference/Journal, Year, URL
*   **一句话核心思想:**
*   **核心贡献清单 (Key Contributions):**
*   **个人评级:** (Game-Changer / Significant Improvement / Incremental / Niche Application)

**`2_THEORY_DECONSTRUCTION.md`**
*   **动机与问题定义:**
    *   **[质询]** 请直接引用论文摘要或引言中定义核心问题的一句话。
    *   **[质询]** 论文将哪些工作列为Prior Work或Baseline？请列出至少3个，并简述其核心局限性。
*   **方法论解剖:**
    *   **模型拓扑结构:**
        *   **[质询]** 嵌入论文中的模型结构图。这个结构分为几个主要的Stage？每个Stage输入和输出的特征图分辨率和通道数是多少？
    *   **关键模块详解:**
        *   **[质询]** 嵌入该模块的示意图或公式。
        *   **[质询]** 对关键公式中的每一个符号进行解释，并注明其Tensor Shape。
        *   **[质询]** 分析关键模块的理论计算复杂度，并给出推导过程。
*   **实验分析:**
    *   **[质询]** 请复现论文中的核心结果表格。
    *   **[质询]** 找到一个关键的消融实验（Ablation Study）表格，并分析哪个组件对结果影响最大，并解释原因。

**`3_ENGINEERING_ANALYSIS.md`**
*   **代码依赖与环境分析:**
    *   **[质询]** 列出`requirements.txt`中最核心的Top 3性能相关依赖项。
    *   **[质询]** 代码中是否包含任何自定义C++/CUDA算子（`.cpp`/`.cu`文件）？其作用是什么？
*   **核心技术解耦与映射:**
    *   **[质询]** 针对论文的每一个核心创新点，完成下表：
| 理论创新点 | **代码实现证据 (文件路径+函数/类)** | **静态性能预判** | **依赖的非标库/算子** |
|:---|:---|:---|:---|
| ... | ... | (例如：访存/计算密集, 潜在瓶颈) | ... |
*   **推理流程静态分析:**
    *   **[质询]** 用伪代码或流程图，描述主模型`forward`函数中输入张量的完整变换路径。
    *   **[质询]** 在该路径上，张量的`shape`发生了几次关键变化？在哪些模块变化的？
*   **资源消耗理论分析:**
    *   **[质询]** 模型的总参数量是多少？（来源：论文/README/代码估算）。
    *   **[质询]** 理论上峰值显存占用可能发生在哪个环节？请说明理由。

**`4_CRITIQUE_AND_REFLECTION.md`**
*   **局限性与待解决问题:**
    *   **[质询]** 论文作者在`Conclusion`或`Limitation`一节中承认了哪些不足？
    *   **[质询]** 在这篇论文发表后，有没有后续的工作专门解决了它的某个局限性？请举例并附上链接。
*   **潜在改进方向:**
*   **对我的启发:**

## 5. 工作流建议 (Suggested Workflow)

1.  **初始化 (Initialize):** 在仓库根目录下创建一个 `_TEMPLATE` 目录，存放上述4个空的MD模板文件。
2.  **深度分析 (Deep Dive):** 当需要分析一篇新论文时，复制 `_TEMPLATE` 目录并重命名为新的“卷宗”名称，然后逐一填充4个MD文件。
3.  **横向对比 (Compare):** 当积累了足够多的“卷宗”后，可手动创建 `COMPARISON_*.md` 文件，利用已有的分析数据填充对比矩阵，生成洞察。

## 6. 总结

本需求规格说明书定义了一个旨在深化个人技术理解、培养架构师思维的知识系统。其核心在于通过严格、结构化的分析协议，将零散的知识点转化为系统性的、可追溯的、高价值的个人技术资产。该项目V1.0的需求已明确，可以进入实施阶段。
